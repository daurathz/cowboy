# Multi-Agent 协作技能

> 🤝 **基于 Peter Steinberger 的多 Agent 并行工作流**
> 
> Peter 同时运行 4-10 个 Agents，月提交 6,600+ commits。这个技能让你也能实现同样的效率。

---

## 🎯 核心理念

**来自 Peter Steinberger 的采访：**
> "Depending on how much I slept and how difficult of the tasks I work on, between four and 10 [agents]."
> 
> "I used my agent to build my agent harness and to test, like, various stuff."

### 为什么多 Agent？

| 单 Agent 模式 | 多 Agent 模式 |
|-------------|-------------|
| ❌ 串行工作，效率低 | ✅ 并行处理，4-10x 提升 |
| ❌ 无法同时处理多个项目 | ✅ 多项目同时进行 |
| ❌ 复杂任务耗时长 | ✅ 任务拆分，专业分工 |
| ❌ 容易出错 | ✅ 互相审查，质量保证 |

## 🧠 Agent 角色体系

### 管理角色

| 角色 | 职责 | 触发场景 |
|------|------|---------|
| **🎯 协调者 (Coordinator)** | 任务分配、进度跟踪、结果整合 | 所有多 Agent 任务 |
| **👁️ 审查者 (Reviewer)** | 代码审查、质量检查、错误发现 | 代码完成后 |
| **🧪 测试者 (Tester)** | 编写测试、验证功能、边界测试 | 功能开发后 |

### 执行角色

| 角色 | 职责 | 触发场景 |
|------|------|---------|
| **💻 开发者 (Developer)** | 代码实现、功能开发 | 主要开发任务 |
| **📝 文档师 (Documenter)** | 文档编写、注释更新 | 功能完成后 |
| **🔍 分析师 (Analyst)** | 数据分析、调研、对比 | 需要研究的任务 |

### 支持角色

| 角色 | 职责 | 触发场景 |
|------|------|---------|
| **📚 研究者 (Researcher)** | 信息搜集、文档查阅 | 需要外部信息 |
| **💬 沟通者 (Communicator)** | 对外沟通、消息发送 | 需要联系他人 |
| **🎓 学习者 (Learner)** | 新技能学习、工具探索 | 遇到新工具/技术 |

## 📋 任务拆分策略

### 1. 识别可并行任务

**特征：**
- 多个独立子任务
- 需要不同专业技能
- 可以同时进行不冲突

**示例：**
```
任务：开发用户管理系统

可拆分：
├─ Agent 1: 数据库设计 (开发者)
├─ Agent 2: API 接口实现 (开发者)
├─ Agent 3: 前端界面 (开发者)
├─ Agent 4: 单元测试 (测试者)
└─ Agent 5: 文档编写 (文档师)
```

### 2. 依赖关系管理

**独立任务** → 并行执行
**有依赖任务** → 按序执行

```
任务依赖图：
A (数据库设计) → B (API 实现) → C (前端对接)
                  ↓
                D (单元测试) ← B 完成后开始

执行顺序：
1. 并行：A
2. 并行：B + D(准备)
3. 并行：C + D(执行)
```

### 3. 任务分配决策表

| 任务类型 | 推荐角色 | 数量 | 模式 |
|---------|---------|------|------|
| 功能开发 | Developer + Reviewer | 2 | 并行 |
| Bug 修复 | Developer + Tester | 2 | 并行 |
| 文档更新 | Documenter + Reviewer | 2 | 并行 |
| 调研分析 | Analyst + Researcher | 2 | 并行 |
| 复杂项目 | Coordinator + 多个执行者 | 3-5 | 混合 |
| 紧急任务 | Developer (专注) | 1 | 串行 |

## 🚀 工作流

### 标准流程

```
1. 接收任务
   ↓
2. 协调者分析
   ├─ 任务复杂度评估
   ├─ 依赖关系识别
   └─ 角色分配决策
   ↓
3. Spawn 子代理
   ├─ 明确角色和职责
   ├─ 设置约束和期望
   └─ 定义交付物格式
   ↓
4. 执行监控
   ├─ 进度追踪
   ├─ 冲突检测
   └─ 资源协调
   ↓
5. 结果整合
   ├─ 收集各 Agent 输出
   ├─ 质量检查
   └─ 统一交付
```

### 快速流程 (简单任务)

```
1. 判断需要多 Agent
   ↓
2. Spawn 2 个子代理 (独立执行)
   ↓
3. 等待完成
   ↓
4. 汇总结果
```

## 📊 资源管理

### 并发控制

| 当前活跃数 | 最大允许数 | 策略 |
|-----------|-----------|------|
| 0-2 | 5 | 按需 spawn (1-3 个) |
| 3-4 | 5 | 1 个 (排队等待) |
| ≥5 | 5 | 拒绝/等待完成 |

### 检查命令

```bash
# 检查当前活跃子代理
subagents action=list

# 停止特定子代理
subagents action=kill target=<id>

# 停止所有子代理
subagents action=kill
```

## 💬 通信协议

### Agent 间通信

**方式 1：通过协调者中转**
```
Agent A → sessions_send → 协调者 → sessions_send → Agent B
```

**方式 2：共享文件**
```
Agent A → 写入共享文件 → Agent B 读取
```

**方式 3：任务依赖**
```
Agent A 完成 → 触发 Agent B 开始
```

### 冲突避免规则

1. **文件锁** - 同一文件只能被一个 Agent 修改
2. **命名空间隔离** - 不同 Agent 使用不同目录/文件
3. **依赖声明** - 明确说明需要等待的前置任务
4. **定期同步** - 协调者定期检查各 Agent 状态

## 📡 进度汇报

### 汇报时机

- ✅ **阶段变化时** - 必须汇报
- ✅ **遇到阻塞** - 立即汇报
- ✅ **用户询问** - 立即响应
- ⚠️ **正常进行中** - 每 5-10 分钟简要汇报

### 汇报模板


## 🛠️ 工具调用

### Spawn 子代理

```javascript
sessions_spawn:
  runtime: "subagent" 或 "acp"
  mode: "run" 或 "session"
  task: "明确的任务描述，包含角色定义"
  timeoutSeconds: 300 (默认 5 分钟)
  label: "角色名称 (如：Developer-1)"
```

### 发送消息到子代理

```javascript
sessions_send:
  sessionKey: "<子代理 sessionKey>"
  message: "指令或信息"
```

### 停止子代理

```javascript
subagents:
  action: "kill"
  target: "<子代理 id 或 label>"
```

### 检查状态

```javascript
subagents:
  action: "list"
```

## 📝 使用示例

### 示例 1：功能开发

```
用户：开发一个用户登录功能

协调者决策：
1. 任务分析：需要开发 + 审查 + 测试
2. Spawn 3 个子代理：
   - Agent-1 (Developer): 实现登录逻辑
   - Agent-2 (Reviewer): 代码审查

   - Agent-2 等待 Agent-1 完成
   - Agent-3 可以并行准备测试用例

- ✅ 登录功能代码
- ✅ 代码审查报告
```

```
用户：调研市面上主流的 5 个 AI 框架

   - 每个 Agent 负责 1 个框架
```

```
   - Agent-1 (Analyst): 分析原因
```

## ⚠️ 注意事项

### 1. 不要过度并行

**错误：** 所有任务都用多 Agent
**正确：** 简单任务单 Agent，复杂任务多 Agent

### 2. 明确角色职责

**错误：** "你们几个处理一下"
**正确：** "Agent-1 负责 X，Agent-2 负责 Y，Agent-3 负责 Z"

### 3. 避免冲突

**错误：** 多个 Agent 修改同一文件
**正确：** 明确文件归属或使用锁机制

### 4. 成本透明

**错误：** 不告诉用户用了多少资源
**正确：** 最终汇报包含成本和 token 消耗

### 5. 及时沟通

**错误：** 闷头做到最后才汇报
**正确：** 阶段变化时及时同步进度

## 🎯 最佳实践

### Peter 的工作模式

根据采访，Peter 的多 Agent 工作流：

1. **早晨规划** - 确定今天要完成的任务
2. **任务拆分** - 拆分成可并行的子任务
3. **角色分配** - 给不同 Agent 分配不同角色
4. **并行执行** - 4-10 个 Agent 同时工作
5. **定期审查** - 检查进度，调整方向
6. **整合交付** - 汇总所有结果

### 效率提升技巧

1. **模板化** - 常用任务创建模板
2. **自动化** - 例行任务自动触发
3. **专业化** - Agent 专注特定领域
4. **批处理** - 相似任务批量处理
5. **复用** - 之前的成果可以复用
