# DK-Subagent Orchestration Skill

_高效管理子代理，处理复杂/耗时/多源验证任务_

---

## 🎯 适用场景

**核心场景：** 当主任务需要**大量并行/耗时的信息收集工作**时，spawn 子代理来处理

**触发条件（双模式）：**

### 显式指定
用户明确表达使用子代理的意图：
- "用子代理处理"
- "spawn 个子代理"
- "后台跑一下"
- "你慢慢查，不着急"
- 指定 timeout/depth 等参数

### 自主判断
AI 根据任务特征自主决策，满足以下任一条件即触发：
- **预计耗时 > 3 分钟**
- **需要多源交叉验证** (≥3 个信息源)
- **任务可并行拆分** (独立子任务 ≥2 个)
- **深度调研类任务** (非简单查询)

---

## 🧠 核心逻辑链

```
用户请求
   ↓
判断是否需要子代理？
   ├─ 否 → 直接处理
   └─ 是 → 继续
         ↓
   检查环境容量
   ├─ 当前活跃子代理数 (subagents list)
   └─ 决策本次 spawn 数量
         ↓
   Spawn 子代理
   ├─ 明确任务目标
   ├─ 设置约束 (timeout/depth/format)
   └─ 定义交付物格式
         ↓
   执行监控
   ├─ 进度汇报 (阶段变化时)
   ├─ 错误处理 (重试机制)
   └─ 用户打断监听
         ↓
   汇总交付
   ├─ 整合子代理结果
   ├─ 说明执行状态
   └─ 附上成本估算
```

## 📊 子代理数量控制

### 环境感知
```bash
# 检查当前活跃子代理
subagents action=list
```

### 数量决策表
| 当前活跃数 | 最大允许数 | 本次 spawn 策略 |
|-----------|-----------|----------------|
| 0-2       | 5         | 按需 (1-3 个)   |
| 3-4       | 5         | 1 个 (排队)     |
| ≥5        | 5         | 0 个 (拒绝/等待) |

**超限处理：**
- 告知用户当前负载情况
- 提供选项：等待/取消/降级处理

---

## 📡 进度汇报机制

### 汇报触发条件
- ✅ **阶段变化时** - 进入新阶段必须汇报
- ⚠️ **无实质进展** - 下一个 5 分钟周期若仍在同一阶段，可跳过汇报
- ❌ **遇到问题** - 立即汇报（触发重试机制时）

### 汇报间隔
- 基础检查频率：每 5 分钟
- 实际汇报：根据进展动态决定

### 汇报内容模板
```markdown
**子代理进度汇报**

📍 当前阶段：[阶段名称]
⏱️ 已耗时：X 分钟
📈 进度：约 XX%

**本阶段完成：**
- [具体成果 1]
- [具体成果 2]

**下一步：**
- [即将进行的工作]

**阻塞点：** (如有)
- [问题描述]
```

## 🔄 错误处理流程

### 重试机制
```
子代理遇到问题
    ↓
自动重试 (最多 3 次)
    ├─ 重试成功 → 继续执行
    └─ 重试失败 → 通知主代理
```

### 主代理评估
```
收到子代理失败通知
    ↓
评估任务重要性
    ├─ 关键任务受阻 → 停掉本次任务，说明原因
    └─ 非关键任务 → 降级处理，继续完成其他部分
```

### 常见错误及处理
| 错误类型 | 重试策略 | 降级方案 |
|---------|---------|---------|
| 接口限流 | 指数退避重试 | 换信息源/跳过 |
| 超时 | 延长 timeout 重试 | 部分交付 |
| 信息不可用 | 换源重试 | 标注缺失 |
| 子代理崩溃 | 重新 spawn | 主代理接手 |

## 👂 用户打断机制

### 监听关键词
- "停" / "停下" / "别做了"
- "够了" / "算了" / "不要了"
- "取消" / "终止"

### 处理流程
```
用户表达停止意图
    ↓
立即停止子代理 (subagents action=kill)
    ↓
清理未完成的任务
    ↓
汇总已完成的部分
    ↓
向用户说明：
- 已停止
- 完成了哪些部分
- 有什么可用成果
```

## 🔗 子代理协作机制

### 独立模式（默认）
- 各子代理独立执行
- 最后统一汇总

### 协作模式（按需）
当任务需要共享中间结果时：
```
子代理 A → 共享存储 → 子代理 B
    ↓                      ↓
  中间结果              读取后继续
```

**实现方式：**
- 通过主代理中转 (`sessions_send`)
- 共享文件/工作区
- 明确依赖关系后按序 spawn

## 💰 成本感知

### 最终汇报必须包含
```markdown
## 执行摘要

**状态**: ✅ 完成 / ⚠️ 部分完成 / ❌ 失败  
**总耗时**: X 分钟  
**子代理数**: N 个  
**估算 Token 消耗**: ~XXK tokens  
**估算成本**: ¥X.XX (按模型费率)
```

### Token 估算方法
- 子代理汇报中记录 `usage` 字段
- 主代理汇总累加
- 按当前模型费率换算

## 🛠️ 工具调用参考

### 检查子代理状态
```
subagents action=list
```

### Spawn 子代理
```
sessions_spawn:
  runtime: "subagent" 或 "acp"
  mode: "run" 或 "session"
  task: "明确的任务描述"
  timeoutSeconds: 300 (默认 5 分钟)
  agentId: [指定 agent]
```

### 发送进度到主会话
```
sessions_send:
  sessionKey: "main"
  message: "进度汇报内容"
```

### 停止子代理
```
subagents:
  action: "kill"
  target: [子代理 id]
```

---

## ⚠️ 注意事项

1. **不要轮询** - 避免频繁调用 `subagents list` 检查状态
2. **汇报精简** - 无实质进展时跳过汇报，避免打扰
3. **错误透明** - 所有问题和处理过程必须在最终结论中说明
4. **成本透明** - 让用户知道"这活儿值不值"
5. **随时可停** - 用户打断时立即响应，不纠缠

## 📝 使用示例

### 示例 1：显式指定
```
用户：用子代理帮我查一下 2024 年 AI 大模型的主要进展，要详细的

AI: 
1. 判断：显式指定 + 深度调研 → 使用子代理
2. 检查容量：当前 0 个活跃 → 可 spawn
3. Spawn: sessions_spawn(task="调研 2024 年 AI 大模型进展", timeout=600)
4. 监控：每 5 分钟检查进展
5. 汇总：整合结果 + 成本说明
```

### 示例 2：自主判断
```
用户：帮我对比一下 SearXNG、Whoogle 和 MetaGear 这三个搜索引擎的优缺点

AI:
1. 判断：多源验证 (≥3 个产品) + 预计耗时 >3 分钟 → 使用子代理
2. 检查容量：当前 1 个活跃 → 可再 spawn 1 个
3. Spawn: sessions_spawn(task="对比三个搜索引擎...", timeout=300)
4. 汇报：阶段变化时汇报进度
5. 汇总：结构化对比表 + 执行说明
```

### 示例 3：用户打断
```
[子代理执行中 3 分钟]
用户：算了，先停一下吧

AI:
1. 监听：识别停止意图
2. 执行：subagents action=kill
3. 汇总：已完成的部分成果
4. 回复："已停止。目前完成了 XX 部分，结果是... 剩余 XX 部分未完成。"
```
