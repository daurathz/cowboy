## 🎯 核心观点与主张

### 观点 1：Agentic Engineering 不是工具升级，是工程范式转变

**主张：**
- 从"人写代码"到"人设计系统，AI 写代码"
- 工程师的核心价值从编码能力转向系统设计能力和验证能力
- 组织需要重新定义工程流程和角色分工

**证据：**
- OpenAI Codex 团队 100 万行代码零人工编写
- Stripe Minions 每周 1000+ PRs 由 agents 合并
- 传统测试范式被 JiTTesting 取代

### 观点 2：上下文工程是核心竞争力

**主张：**
- 上下文质量决定 AI 输出质量
- 上下文管理是新的"编程技能"
- 组织需要建立上下文工程规范和工具

**证据：**
- John Kim: "我花更多时间管理上下文而不是写提示词"
- 失败案例：把一切丢进上下文希望 AI 理解
- 成功案例：懒加载、分层 CLAUDE.md、保持精简

### 观点 3：验证循环比一次性正确更重要

**主张：**
- 没有验证循环的 AI coding 是"希望与祈祷"
- 验证能力是 agentic engineering 的第一技能
- 组织需要投资自动化验证基础设施

**证据：**
- "AI coding 对我没用"的人 9/10 没有验证循环
- Meta 的 JiTTesting 专为 agentic 开发设计
- TELUS 节省 50 万小时来自结构化验证

### 观点 4：多 agent 编排是规模化关键

**主张：**
- 单 agent 适合个人生产力，多 agent 适合组织规模化
- 专业化 agent 团队 (作者、测试、审查、安全) 是必然方向
- 组织需要定义 agent 角色和协作协议

**证据：**
- Stripe Minions: 任务→实现→CI→PR→合并 全流程自动化
- Zapier: 800+ agents 内部部署
- NxCode: 多 provider agent pipelines (Claude 推理 + Codex 实现)

### 观点 5：安全必须嵌入第一天

**主张：**
- Agent 规模开发需要 Agent 规模安全
- 手动安全审查跟不上 agent 生成速度
- 安全质量门是每个 PEV 循环的必需部分

**证据：**
- Anthropic 2026 报告：agents 可以大规模引入漏洞
- 1000 PRs/周 × 1% 漏洞率 = 每周 10 个新漏洞
- 自动化防御是必需的

### 二、流程规范 (Process)

#### 2.1 PEV 工作流

```
┌─────────────────────────────────────────────────────────────┐
│                        PEV 循环                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  PLAN (计划)                                                │
│  ├── 定义目标：验收标准是什么？                             │
│  ├── 任务分解：拆分为 agent 大小的工作单元                   │
│  ├── 设置约束：架构边界、技术栈规则、风格指南               │
│  ├── 指定质量门：测试、审查、安全检查                       │
│  └── 分配角色：哪个 agent 负责什么？                        │
│                                                             │
│  ↓                                                          │
│                                                             │
│  EXECUTE (执行)                                             │
│  ├── Feature Author: 写代码                                 │
│  ├── Test Generator: 写测试                                 │
│  ├── Code Reviewer: 审查代码                                │
│  ├── Architecture Guardian: 检查架构合规                    │
│  ├── Security Scanner: 漏洞扫描                             │
│  └── Documentation Writer: 更新文档                         │
│                                                             │
│  ↓                                                          │
│                                                             │
│  VERIFY (验证)                                              │
│  ├── 满足验收标准吗？                                       │
│  ├── 引入安全漏洞吗？                                       │
│  ├── 架构一致吗？                                           │
│  ├── 测试有意义吗？                                         │
│  └── 人类会批准这个 PR 吗？                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 2.2 代码审查流程

```
Agent 生成代码
    ↓
自动验证 (测试 + lint + 安全扫描)
    ↓
通过？──否──→ Agent 修复 (循环)
    │
   是
    ↓
人类审查 (重点：架构、安全、业务逻辑)
    ↓
批准？──否──→ 反馈 + Agent 修复
    │
   是
    ↓
合并 + 部署
```

#### 2.3 上下文管理规范

**CLAUDE.md 结构 (约 300 行):**
```markdown
# 项目概述
- 这是什么项目
- 核心功能
- 技术栈

# 领域知识
- 业务规则
- 关键概念
- 数据模型

# 验证命令
- 构建：npm run build
- 测试：npm test
- Lint: npm run lint
- 安全：npm run security-scan

# 关键规则 (最高优先级)
- 规则 1
- 规则 2
- ...

# 工作流触发器
- "构建应用" → 运行构建脚本
- "添加功能" → 遵循功能开发流程
```

**懒加载原则:**
- 根目录：索引 + 核心规则
- 子目录：详细文档
- Agent 按需加载

### 三、规则体系 (Rules)

#### 3.1 代码生成规则

| 规则 | 描述 | 执行方式 |
|------|------|----------|
| **测试先行** | 新功能必须先有测试 | CLAUDE.md 强制 |
| **验证循环** | 必须有构建/测试/修复循环 | Agent 工作流 |
| **小步提交** | 每次变更 <500 行 | Git hooks |
| **文档同步** | 代码变更必须更新文档 | 审查检查点 |
| **安全扫描** | 每次 PR 必须通过安全扫描 | CI 质量门 |

#### 3.2 上下文使用规则

| 规则 | 描述 | 执行方式 |
|------|------|----------|
| **保持精简** | CLAUDE.md ~300 行 | 定期审查 |
| **懒加载** | 不预加载一切 | 目录结构 |
| **定期清理** | 使用 `/clear` 重置 | Agent 提醒 |
| **审计 token** | 定期 `/context` 检查 | 心跳检查 |

#### 3.3 安全规则

| 规则 | 描述 | 执行方式 |
|------|------|----------|
| **权限最小化** | Agent 只有所需权限 | 权限配置 |
| **危险命令拦截** | PreToolUse hooks 阻止 | Hooks 配置 |
| **密钥管理** | 不硬编码密钥 | 代码审查 + SAST |
| **依赖审查** | 新依赖需审批 | CI 检查 |

### 四、组织形式 (Organization)

#### 4.1 Agent 团队结构

```
┌─────────────────────────────────────────────────────────┐
│                    人类工程师 (Architect)                │
│                   - 系统设计                            │
│                   - 意图指定                            │
│                   - 最终验证                            │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│                   Agent 协调器 (Orchestrator)            │
│                   - 任务分解                            │
│                   - Agent 分配                          │
│                   - 进度跟踪                            │
└─────────────────────────────────────────────────────────┘
                          ↓
    ┌───────────┬───────────┬───────────┬───────────┐
    ↓           ↓           ↓           ↓           ↓
┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐
│ Feature│ │  Test  │ │  Code  │ │Security│ │  Docs  │
│ Author │ │Generator│ │Reviewer│ │Scanner │ │ Writer │
└────────┘ └────────┘ └────────┘ └────────┘ └────────┘
```

#### 4.2 人类角色转变

| 传统角色 | Agentic 角色 | 核心职责 |
|----------|-------------|----------|
| 初级工程师 | Agent 训练师 | 训练 agent 理解代码库 |
| 中级工程师 | Agent 协调器 | 编排多 agent 工作流 |
| 高级工程师 | 系统架构师 | 设计约束系统和验证流程 |
| Tech Lead | 质量负责人 | 定义质量门和审查标准 |
| 经理 | 生产力负责人 | 优化 agent 效率和指标 |

#### 4.3 团队协作模式

**同步工作:**
- 多终端实例并行 (iTerm split panes)
- 通知启用 (完成提示音)
- Git worktrees 隔离文件

**异步工作:**
- 任务发布到 Agent 协调器
- Agent 自主执行 PEV 循环
- 人类在检查点审查

**知识传承:**
- CLAUDE.md 版本控制
- Skills 库共享
- 会话学习持久化

### 五、度量指标 (Metrics)

#### 5.1 生产力指标
- **Agent 成功率**: 一次性通过审查的 PR 比例
- **审查拒绝率**: 人类审查拒绝的 PR 比例
- **Bug 引入率**: 生产环境 bug 中 agent 引入的比例
- **交付速度**: 功能从需求到上线的时间

#### 5.2 质量指标
- **测试覆盖率**: 自动生成的测试覆盖率
- **安全漏洞数**: SAST 扫描发现的漏洞数
- **技术债务**: 代码复杂度、重复度趋势
- **文档同步率**: 代码变更后文档更新比例

#### 5.3 效率指标
- **上下文 token 使用**: 平均每次会话 token 消耗
- **验证循环次数**: 平均每个功能需要多少次验证循环
- **Agent 利用率**: Agent 活跃时间比例

## ⚠️ 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Agent 生成低质量代码 | 技术债务增加 | 严格质量门 + 人类审查 |
| 安全漏洞 | 数据泄露风险 | 自动化 SAST + 安全扫描 |
| 过度依赖 AI | 人类技能退化 | 定期代码审查 + 培训 |
| 上下文污染 | AI 输出质量下降 | 懒加载 + 定期清理 |
| Agent 失控 | 意外变更 | 权限控制 + Git 安全网 |
