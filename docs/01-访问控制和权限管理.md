# 访问控制和权限管理

> 钉钉场景下的企业 Agent 权限控制方案

---

## 🎯 核心挑战

```
钉钉用户 → Agent → 公司 API
   ↓         ↓         ↓
 身份？   权限？   范围？
```

如何保证不同岗位的同事通过钉钉与 Agent 沟通时，Agent 只做该用户及岗位权限范围内的事？

---

## ✅ 解决方案：三层身份映射 + 动态权限注入

```
┌─────────────────────────────────────────────────────────┐
│  身份层：钉钉用户 → 企业内部用户 → Agent 会话身份        │
├─────────────────────────────────────────────────────────┤
│  权限层：岗位角色 + 数据范围 + 时间/环境约束            │
├─────────────────────────────────────────────────────────┤
│  执行层：API 调用前鉴权 + 调用中监控 + 调用后审计        │
└─────────────────────────────────────────────────────────┘
```

---

## 1️⃣ 身份映射机制

### 钉钉身份认证流程

```yaml
步骤 1: 钉钉 OAuth 2.0
  - 用户发送消息到 Agent
  - 钉钉回调携带用户信息 (union_id, department, etc.)
  - 验证回调签名 (防伪造)

步骤 2: 映射企业内部用户
  - 根据 union_id 查询 HR 系统/AD/LDAP
  - 获取用户岗位、职级、部门信息
  - 获取用户数据权限范围

步骤 3: 生成 Agent 会话 Token
  - 签发 JWT Token (有效期 1 小时)
  - Token 包含完整用户上下文
  - Token 绑定设备指纹 + IP
```

### JWT Token 结构示例

```json
{
  "user_id": "zhangsan",
  "dingtalk_union_id": "dt_xxxxx",
  "department": "研发部",
  "role": "高级工程师",
  "data_scope": ["project_a", "project_b"],
  "permissions": ["api:read", "api:write"],
  "exp": 1772629200,
  "iat": 1772625600
}
```

## 2️⃣ 权限控制逻辑链

### 完整流程

```
用户提问
   ↓
提取意图 (要调用哪个 API？访问什么数据？)
   ↓
权限检查 (ABAC 策略引擎)
   ├─ 用户属性：岗位/职级/部门
   ├─ 资源属性：数据敏感度/所属项目
   ├─ 环境属性：工作时间/IP 范围/设备
   └─ 行为属性：请求频率/历史行为
   ↓
决策结果
   ├─ 允许 → 注入权限上下文 → 执行 API 调用
   ├─ 拒绝 → 返回友好提示
   └─ 需审批 → 触发审批流 → 等待结果
```

### ABAC 策略示例

```yaml
# 策略 1：研发人员访问项目数据
rule: "allow_project_data_access"
when:
  - user.department in ["研发部", "产品部"]
  - user.role in ["高级工程师", "技术专家"]
  - resource.project in user.data_scope
  - context.time between "09:00" and "18:00"
  - context.location in ["公司网络", "VPN"]

# 策略 2：财务数据仅限财务人员
rule: "allow_finance_data_access"
when:
  - user.department == "财务部"
  - resource.sensitivity == "internal"
  - action in ["read", "query"]  # 禁止 write/delete

# 策略 3：敏感操作需审批
rule: "require_approval_for_sensitive"
when:
  - resource.sensitivity == "confidential"
  - action in ["delete", "export", "share"]
then:
  - trigger_approval_flow
  - wait_for_manager_approval
```

### 权限检查代码示例

```python
# 伪代码示例
async def check_permission(user_context, action, resource):
    # 加载用户策略
    policies = await load_policies(user_context.department, user_context.role)
    
    # ABAC 引擎评估
    result = abac_engine.evaluate(
        subject=user_context,
        action=action,
        resource=resource,
        environment=get_context()
    )
    
    if result.allowed:
        # 注入权限上下文
        return inject_permission_context(result)
    elif result.requires_approval:
        # 触发审批
        return await trigger_approval(result)
    else:
        # 拒绝访问
        raise PermissionDenied(result.reason)
```

## 3️⃣ 钉钉集成关键点

### 消息入口安全

```yaml
回调验证:
  - 验证钉钉签名 (signature)
  - 验证时间戳 (防重放攻击)
  - 验证消息来源 IP (钉钉服务器 IP 白名单)

会话管理:
  - 会话 ID 与钉钉会话绑定
  - 防止会话劫持
  - 会话超时自动失效
```

### 多 IM 平台支持

```yaml
统一抽象层:
  钉钉 OAuth → 统一身份接口 ← 企业微信 OAuth
  飞书 OAuth →              ← 其他 IM 平台
```

## 4️⃣ 权限模型对比

| 模型 | 适用场景 | 优点 | 缺点 |
|-----|---------|------|------|
| **RBAC** | 稳定权限结构 | 简单直观、易管理 | 角色爆炸、灵活性差 |
| **ABAC** | 动态权限场景 | 细粒度、灵活 | 策略复杂、难调试 |
| **ReBAC** | 关系型权限 | 自然表达关系 | 实现复杂 |
| **PBAC** | 统一管理 | 策略与代码解耦 | 需要策略引擎 |

### 推荐：RBAC + ABAC 混合模型

```yaml
RBAC 层:
  - 定义基础角色 (管理员/普通用户/访客)
  - 角色映射到权限组
  
ABAC 层:
  - 在角色基础上增加动态约束
  - 时间/地点/数据范围等条件
  
PBAC 实现:
  - 使用策略引擎 (Oso/OPA) 统一管理
  - 策略可独立审计和版本控制
```

## 5️⃣ 实际案例

### 案例 1：研发人员访问项目数据

```
用户：张三 (研发部 - 高级工程师)
请求：查看项目 A 的代码仓库

权限检查:
  ✓ 部门匹配 (研发部)
  ✓ 角色匹配 (高级工程师)
  ✓ 项目在数据范围内 (project_a)
  ✓ 工作时间 (14:30)
  ✓ 公司网络

结果：允许访问
```

### 案例 2：跨部门访问被拒绝

```
用户：李四 (市场部 - 经理)
请求：查看项目 B 的技术文档

权限检查:
  ✗ 部门不匹配 (市场部 vs 研发部)
  ✗ 项目不在数据范围内

结果：拒绝访问
提示："抱歉，您没有权限访问该项目文档。如需访问，请联系项目负责人。"
```

### 案例 3：敏感操作触发审批

```
用户：王五 (研发部 - 技术专家)
请求：导出客户数据

权限检查:
  ✓ 部门匹配
  ✓ 角色匹配
  ⚠ 数据敏感度：机密
  ⚠ 操作类型：导出 (需审批)

结果：触发审批流程
通知：已提交审批，等待部门经理批准
```

## 6️⃣ 最佳实践

### ✅ 应该做的

- [ ] 最小权限原则 (只授予必要的权限)
- [ ] 定期权限审查 (每季度)
- [ ] 完整审计日志 (谁/何时/做了什么)
- [ ] 异常行为检测 (频率/时间/内容异常)
- [ ] 权限变更通知 (用户权限变化时通知)

### ❌ 不应该做的

- [ ] 硬编码权限检查逻辑
- [ ] 在前端/Agent 层信任用户传递的身份
- [ ] 长期有效的 Token
- [ ] 明文存储敏感权限信息
- [ ] 忽略审计和监控
