# 4. AI Coding 标准与最佳实践

## 4.1 AI 辅助开发规范

### 代码生成原则

1. **Spec First（规范优先）**
   - 禁止直接让 AI 写代码
   - 必须先有 spec.md，包含：需求、架构、数据模型、测试策略
   - 规划时间：项目总时间的 15-20%

2. **Small Chunks（小步迭代）**
   - 每次只让 AI 实现一个功能点/函数
   - 禁止一次性生成整个模块
   - 每步完成后必须测试

3. **Context Rich（丰富上下文）**
   - 提供相关代码、API 文档、约束条件
   - 使用工具：gitingest、Context7、Repo2TXT
   - 明确告知"不要做什么"

4. **Human Review（人工审查）**
   - AI 生成的所有代码必须人工审查
   - 重点审查：业务逻辑、边界条件、安全考量
   - 审查清单标准化

### CLAUDE.md 规范（项目级 AI 配置）

每个项目根目录创建 `CLAUDE.md` 或 `.cursorrules`：

```markdown
# 项目 AI 助手配置

## 项目概述
[简要描述项目目的、技术栈、架构]

## 编码规范
- 语言：TypeScript 5.x, React 18, Node.js 20
- 代码风格：遵循 .eslintrc 配置
- 命名约定：camelCase 变量，PascalCase 组件

## 架构约束
- 禁止：循环依赖、全局状态
- 推荐：依赖注入、函数式编程

## 测试要求
- 单元测试覆盖率：>80%
- 必须为公共函数编写测试
- 使用 Jest + React Testing Library

## 安全考量
- 禁止：硬编码密钥、SQL 拼接
- 必须：输入验证、XSS 防护

## 常见陷阱
- [列出项目特定陷阱和解决方案]

## 有用命令
- `npm run test`: 运行测试
- `npm run lint`: 代码检查
```

## 4.2 Code Review 流程变革

### 传统 Review vs AI 增强 Review

| 环节 | 传统流程 | AI 增强流程 |
|------|----------|-------------|
| **初筛** | 人工浏览变更 | AI 自动审查（安全、规范、测试） |
| **深度审查** | 人工逐行审查 | AI 标记高风险区域，人工聚焦 |
| **测试验证** | 人工运行测试 | AI 生成补充测试 + 自动运行 |
| **反馈循环** | 人工写评论 | AI 草拟评论，人工修改发送 |

```
请审查以下代码变更：

## 变更描述
[功能说明]

## 审查重点

- ✅ 通过项
- ⚠️ 警告项（需关注但不阻塞）

- 💡 改进建议（可选优化）
```

# 8. 风险与应对

## 8.1 主要风险识别

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|----------|
| **抵触情绪** | 高 | 高 | 早期沟通、试点示范、激励机制 |
| **过度依赖 AI** | 中 | 高 | 人工审查强制、能力评估、轮岗 |
| **质量下降** | 中 | 高 | 质量门禁、AI 审查 + 人工双保险 |
| **安全风险** | 中 | 极高 | 安全培训、自动化扫描、审计日志 |
| **成本超支** | 中 | 中 | 用量监控、预算预警、优化建议 |
| **工具锁定** | 低 | 中 | 模块化设计、开源备选、数据导出 |
| **人才流失** | 低 | 高 | 职业发展路径、技能认证、激励机制 |

## 8.2 关键风险详解

### 风险 1: 抵触情绪（尤其是资深工程师）

**表现**:
- "AI 写的代码不可靠"
- "我不用也能完成工作"
- "这是在取代我们"

**应对**:
1. **高层背书**: CTO/技术 VP 公开支持，明确转型是"增强"而非"取代"
2. **试点示范**: 选择受尊重的资深工程师参与试点，产出成功案例
3. **数据说话**: 展示试点团队效率提升、工作满意度提升数据
4. **激励机制**: AI 使用纳入绩效、设立"AI 先锋"奖项
5. **渐进式**: 允许逐步采用，不强制一刀切

### 风险 2: 过度依赖 AI 导致能力退化

**表现**:
- 离开 AI 不会写代码
- 无法理解 AI 生成的复杂逻辑
- 调试能力下降

**应对**:
1. **强制人工审查**: AI 生成代码必须人工审查并理解
2. **定期"无 AI 日"**: 每周 1 天不使用 AI，保持基础能力
3. **能力评估**: 季度技能评估，包含无 AI 场景测试
4. **轮岗机制**: 核心模块定期轮换负责人
5. **知识沉淀**: 强制文档化关键逻辑和设计决策

### 风险 3: 安全风险增加

**表现**:
- AI 生成代码包含安全漏洞
- 敏感数据泄露风险
- 依赖库漏洞增加

**应对**:
1. **安全培训**: 全员 AI 安全培训，考核通过才能使用
2. **自动化扫描**: CI/CD 集成 SAST、依赖扫描、密钥检测
3. **审查清单**: AI 代码审查必须包含安全检查项
4. **审计日志**: 记录 AI 使用日志，定期审计
5. **数据隔离**: 禁止将敏感代码/数据上传到公共 AI 服务

## 8.3 应急预案

### 预案 1: 工具服务中断

**场景**: Claude Code/Cursor 等服务不可用

**应对**:
1. **本地备选**: 部署本地模型（Ollama + CodeLlama）
2. **多工具策略**: 不依赖单一工具，保持 2-3 个备选
3. **离线能力**: 确保核心开发可在离线环境进行
4. **降级流程**: 明确服务中断时的降级开发流程

### 预案 2: 大规模质量问题

**场景**: AI 生成代码导致生产事故

**应对**:
1. **快速回滚**: 确保一键回滚能力
2. **根因分析**: 分析是 Prompt 问题、模型问题还是审查疏漏
3. **流程加固**: 针对根因优化审查流程
4. **案例学习**: 全员学习，避免重复
